# 局部微调：修正实现与设计稿的差异

你将对比 Figma 设计稿与当前代码实现，找出差异并进行精准的局部修正。不重新生成整个组件，只修改有偏差的部分。

## 第一步：确认修正范围

向用户确认：

1. 哪个文件/组件需要修正？
2. 具体哪些方面有偏差？（如果用户没说明，则全面对比）
   - 间距（padding/margin/gap）
   - 颜色（背景色/文字色/边框色）
   - 字体（大小/粗细/行高/字间距）
   - 布局（对齐/方向/换行）
   - 尺寸（宽高/圆角）
   - 其他视觉属性（阴影/透明度/边框）

## 第二步：获取设计数据

1. 调用 `get_node` 获取目标节点：
   - `format: "json"` + `precision: "pixel-perfect"` — 确保属性完整
   - `depth: 15` — 包含所有子节点
2. 调用 `get_node_css` 获取精确样式（使用项目对应的 css 或 tailwind 模式）
3. 如有图片/图标差异，调用 `get_images` 或 `export_svg`

## 第三步：读取当前实现

1. 读取用户指定的组件文件
2. 识别当前使用的样式方案（Tailwind / CSS Modules / styled-components / 内联样式等）
3. 解析当前样式值，建立属性对照表

## 第四步：逐项对比差异

对比以下维度，列出所有偏差：

| 维度 | 设计稿值            | 当前实现值     | 偏差     |
| ---- | ------------------- | -------------- | -------- |
| 间距 | padding/gap 精确 px | 当前代码中的值 | 差异量   |
| 颜色 | hex/rgba            | 当前代码中的值 | 色差     |
| 字体 | size/weight/lh/ls   | 当前代码中的值 | 差异     |
| 布局 | flex 方向/对齐      | 当前代码中的值 | 是否一致 |
| 尺寸 | width/height/radius | 当前代码中的值 | 差异量   |

向用户展示差异表，确认哪些需要修正（如果差异较多）。

## 第五步：精准修正

### 修正原则

- **最小改动**：只修改有偏差的属性，不动其他代码
- **保持风格**：使用项目已有的样式写法，不引入新模式
- **保留逻辑**：不改动组件的交互逻辑、事件处理、状态管理
- **尊重 token**：如果项目有 design token 体系，优先使用 token 而非硬编码值

### 常见修正模式

**Tailwind 项目：**

- 间距偏差 → 替换 `p-4` 为 `p-[18px]` 或找到最近的 token
- 颜色偏差 → 替换 `text-gray-500` 为 `text-[#6B7280]` 或正确的 token
- 字体偏差 → 调整 `text-sm` 为 `text-[13px]`、`leading-5` 为 `leading-[18px]`

**CSS/SCSS 项目：**

- 直接修改对应属性值
- 如果值来自变量，修改变量定义或改用精确值

**CSS-in-JS 项目：**

- 修改样式对象中的对应属性

## 第六步：验证修正

修正完成后，逐项确认：

1. 所有标记的偏差是否已修正？
2. 修正后的值是否与设计稿精确匹配？
3. 是否引入了新的问题（如布局错位）？
4. 代码风格是否与项目一致？

## 输出

完成后简要说明：

- 修正了哪些属性（列出 before → after）
- 哪些差异保留未改（如果有，说明原因）
- 是否有需要手动确认的部分
